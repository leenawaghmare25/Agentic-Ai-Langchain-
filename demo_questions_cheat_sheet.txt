SECTION 1: REPOSITORY ARCHITECTURE & DIRECTORY STRUCTURE
================================================================================

QUESTION 1: "Explain the directory layout. What does each folder and file do?"
★ Easy Explanation:
  - agent.py: The central conductor. It coordinates our 8 specialized agents, registers the tools they use, manages fallbacks, and executes the SDLC cycles.
  - .agent_memory.txt: Acts as persistent long-term preference storage. Any instructions written here are loaded at startup and followed by the developer agents.
  - .env: Stores configuration parameters and secure API keys (like GROQ_API_KEY and GEMINI_API_KEY).
  - requirements.txt: Lists libraries needed to run the app (like langchain, langchain-groq, langchain-ollama, colorama, and duckduckgo-search).
  - projects/: The sandbox directory. When you run agent.py, it prompts you for a project name (e.g. calculator) and generates all frontend/backend files inside a subfolder here, isolating it from the main script.
  - projects/calculator/: A completed example generated by our agent framework, consisting of a Flask backend (app.py) and a web frontend (static/ and templates/ directories).
  - docs/: Generated automatically inside the project folder (e.g. projects/calculator/docs/). It stores permanent artifacts written by the agents (requirements.txt, planning.txt, architecture.txt, qa_report.txt, etc.).

QUESTION 2: "Where does the agent define its sandbox directory, and how does it prevent file access escape?"
★ Easy Explanation:
  At startup, the script sets WORKSPACE_DIR to the root workspace. When the user inputs a project name (like 'calculator'), it updates CURRENT_PROJECT_DIR to point to 'projects/calculator'.
  To ensure the agents cannot delete system files or read sensitive files outside the sandbox, every file tool call goes through a path resolver helper. If the path tries to escape the sandbox (e.g., using relative pathways like "../"), it blocks it immediately and throws a security access error.

QUESTION 7: "What happens if a cloud provider goes down or you hit a rate limit?" (Fallback Ladder)

QUESTION 11: "Explain the architecture of the generated Calculator app."
★ Easy Explanation:
  - Backend (app.py): Spawns a Flask application. It defines two endpoints:
    - '/' (index): Renders the frontend template index.html.
    - '/calculate' (methods=['POST']): Accepts a JSON payload containing the mathematical expression string, evaluates it, and returns the result as JSON.
  - Frontend (templates/index.html & static/style.css): Spawns a beautiful, responsive user interface featuring a calculator grid (buttons for digits 0-9 and operators +, -, *, /, decimal, and clean/calculate controls). Javascript uses fetch() to send the expression to the backend dynamically.

QUESTION 12: "Your backend app.py uses eval(). How did you make it secure against code injection?"
★ Easy Explanation:
  Using eval() directly is dangerous because a user could input Python code (like 'import os; os.system("rm -rf ...")') to hack the server.
  We implemented two major security constraints:
  1. Regular Expression Whitelist: We define a compiled regex pattern (SAFE_PATTERN) that only allows digits, basic operators (+, -, *, /), spaces, decimals, and parentheses. Any request with letters or other symbols is rejected with a HTTP 400 error immediately.
  2. Block Builtins: When calling eval(), we explicitly clear globals and pass {"__builtins__": None}. This prevents the expression from accessing any Python functions or system libraries, blocking all malicious code executions.
★ Code to Show:
  - Point to file: projects/calculator/app.py
  - Regex pattern: Line 6 (SAFE_PATTERN = re.compile(r'^[0-9+\-*/\s().]+$'))
  - Secure evaluation: Line 28
    Code:

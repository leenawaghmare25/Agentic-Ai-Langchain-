MENTOR: "I see a config dictionary here called AGENT_LLM_CONFIG on line 620. Why do some agents use Groq and others use Ollama?"

STUDENT: "We use a hybrid local-and-cloud architecture to optimize performance:
- We map planning and system design agents (like Requirement Analyst and Planning Agent) to cloud models on Groq or Gemini because they require fast, intelligent reasoning to parse user goals.
- We map developer and QA agents (like Frontend Developer, Backend Developer, Integration) to local Ollama models (specifically Qwen 2.5 Coder 7B). This is because code writing requires large token counts, and running them locally avoids paying API costs or hitting API request rate limits."

MENTOR: "Interesting. What happens if Groq is down or you run out of API credits? Does the whole pipeline crash?"

STUDENT: "No, Sir. We built a robust automated recovery system called 'invoke_agent_with_fallback' starting at line 730. 
If a cloud call raises an exception (like rate limits or billing errors), our code catches it (line 755), sleeps 5 seconds to retry (line 792), and if it still fails, it automatically switches the provider to the next option in our fallback ladder list (line 738):
  
  fallback_ladder = ["Gemini", "Groq", "Ollama_7b", "Ollama_1.5b"]

It will keep falling back until it reaches a local Ollama model, ensuring the pipeline never crashes mid-development."

MENTOR: "And how does the output of one agent get passed to the next? How do they chain together?"

STUDENT: "We use a central text accumulator variable called 'pipeline_context' initialized at line 869.

This means Agent 2 gets all information from Agent 1, Agent 3 gets all information from Agents 1 & 2, and so on."

MENTOR: "How do your Frontend and Backend developers write actual files to disk without manual copy-pasting?"

STUDENT: "We register a 'write_file' tool (defined at line 547) and bind it to the LangChain LLM client.
The developer agent runs in a loop (up to 15 cycles, line 998). On each cycle, the agent decides which file to write, calls the 'write_file' tool, and receives a success message back as an 'observation' (line 1058), which gets saved to its chat scratchpad memory (lines 1060-1061) so it remembers what it did in the previous cycles."

MENTOR: "How do you verify that the generated project is functional and doesn't crash on start?"

If any errors are detected, the system automatically runs the agent cycles again to fix them before finishing."

MENTOR: "Let's check the generated app. In 'projects/calculator/app.py', I see you use the eval() function. Isn't eval() dangerous for code injection?"

STUDENT: "Normally yes, Sir, but we secured it using two main techniques:
1. Regex Whitelist (Line 6 in app.py): We compile a SAFE_PATTERN that only allows numbers and basic math signs: [0-9+-*/().\s]. If the input has any letters or security characters, it rejects the request immediately.
2. Clearing Builtins (Line 28 in app.py): When running eval, we pass {"__builtins__": None}. This strips all system access, imports, and file functions from the evaluation scope, rendering it completely safe."
